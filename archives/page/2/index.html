2、node_gyp rebuild卡住

在初始化hexo时会卡到node_gyp rebuild过程,等待很久没有反应。

node_gyp的编译需要python和编译工具：

  • python (v2.7 recommended, v3.x.x is not supported)
  • make
  • A proper C/C++ compiler toolchain, like GCC

如果以上都已经安装则可能是 NodeJs 源码缺失的原因。
node-gyp 编译时候需要 NodeJs 源码来提供头文件，所以它会先尝试下载 NodeJs 源码，由于国外CDN的原因,有时候下载 NodeJs 非常慢的。
那么就自己用迅雷等下载工具下载对应版本的nodejs源码,然后提取给node_gyp.
切记对应的node版本号。我是直接查看本地安装的node版本号下载对应node源码。

注意，不同操作系统下和编译不同的 package 时候 node-gyp 需要的源码版本有可能是不一样的，installVersion 也有可能不一样。推荐先检查一下 node-gyp 的输出日志来确定下载哪一个文件。

另外还可以使用 npm install —loglevel verbose 查看安装时候的详细日志，来确定需要下载的 Nodejs 源码地址。
参考:https://breeswish.org/blog/2014/11/14/node-gyp-rebuild-freeze/

为什么要有Optional

Optinal是swift处理空值的一种方式。不同与Objective-C，swift中的包含nil值的变量是很危险的。如果向一个nil值发送消息，会引起runtime erro导致程序crash。
但是nil值的存在是不可避免的。

所有可能有nil值的变量，都使用Optional变量来兼容nil值和包含实际值的情况。

使用Optional变量

我们知道，Optional变量实际上是对实际对应类型的值的包装(wrapped)。使用时需要对其进行解包(unwrapped)操作，把我们需要的真实的值从Optional中抽取出来。
swift提供了几种解包的方式：

  • Optional Binding 安全的方式
  • Forced Unwrapping 不安全
    如果能够确定变量有值，则可以使用后缀!方式强制解包，取得真实值。但是如果变量为nil，强制解包会引起crash。

Implicitly Unwrapped Optional (隐式解包可选类型)

因为Optional Binding的解包方式太冗长，所以一般情况下，在确定变量有值的情况下，都使用Forced Unwrapping的方式来解包。 在实际的使用中，可选类型的变量可能非常多。这种情况下，强制解包的写法依然显得有些繁琐。为了解决这个问题，swift提供了Implicitly Unwrapped Optional，是一个和Optional不同的enum。与Optional语法糖不同，Implicitly Unwrapped Optional使用’!’代替’?’来实现。
使用Implicitly Unwrapped Optional变量时，默认是已经解绑的值，可以像普通变量一样使用。
Implicitly Unwrapped Optional适合满足两个条件的变量：

  • 创建时是nil值
  • 使用时一定有值的变量，如IBOutlet变量
